  docCommand.vue 深度优化分析

  基于昨天的分析结论 + 当前代码实际情况 + 全局设置系统（pages/my/my.vue
  的配置），以下是完整的优化建议：

  ---
  一、必须改的硬伤（阻塞性问题）

  1. 未接入全局主题系统
  当前使用 styleManager.styleMixin，无法响应 pages/my/my
  中的主题色/深色/字号设置。需要换为 settingsMixin。

  2. 条件编译语法错误 (docCommand.vue:1874-1895)
  // ❌ 当前 — uni-app 不支持 || 语法
  // #ifdef APP-PLUS || APP
  return true;
  // #endif

  // ✅ 正确写法
  // #ifdef APP-PLUS
  return true;
  // #endif
  // #ifdef APP
  return true;
  // #endif

  3. height: 100vh 布局 bug (docCommand.vue:1902)
  App 端 100vh 会被 navigation bar 撑破。需改为 min-height: 100vh +
  padding-bottom。

  4. doc-detail-popup.vue 也未接入主题系统
  弹窗组件同样使用旧 mixin，打开详情时不会跟随主题变化。

  ---
  二、代码结构优化（减少 30%+ 代码量）

  5. 提取通用 clearParams() 方法

  当前 8 处重置逻辑各自手写了清空代码：
  - resetToReinputState (730-767)
  - onCmdClear (821-839)
  - cmdOptChange (852-909)
  - clearAllDataOnCodeTypeSwitch (615-639)
  - submitData.success (1748-1777)
  - cancelParam (1048-1077)
  - cancelNumParam (1079-1088)
  - watch.cmdOpt.handler (451-473)

  6. 精简模板中的 6 种参数渲染变体

  isPushErp、unit、num、普通（非num）、docNo/docType、SN — 可以用统一的
  <param-row> 组件减少 100+ 行模板。

  7. validateNumWithConfirm 拆分 (1145-1278 行，134行)

  当前一个方法做了：获取上下文 → 调API → 解析结果 → 弹窗 → 填值。建议拆为：
  - getNumValidationContext() — 收集 itemSeq / unitId 等
  - applyNumAfterValidation(num, param) — 验证通过后的填值逻辑
  - handleNumValidationError(err, param) — 验证失败处理

  ---
  三、UX 优化

  8. 统一消息反馈通道

  现在混用 3 种：message-box（页面内）、uni.showToast（顶部浮层）、uni.showModal
  （居中弹窗）。体验割裂。

  建议：
  - message-box → 仅用于非阻塞的操作反馈（"已选择: XX指令"、"自动填充了日期"）
  - uni.showToast → 仅用于提交成功/失败（已经被 uni 框架限制，保留）
  - uni.showModal → 仅用于需要用户确认的操作（删除、重置）

  9. 添加操作震动反馈

  pages/my/my.vue 中已有震动开关设置，但 docCommand 完全没有使用：

  // 在扫码成功、提交成功时调用
  triggerHaptic() {
    // #ifdef APP-PLUS
    if (this.appSettings.vibration) plus.device.vibrate(15)
    // #endif
  }

  10. 参数列表增加"当前聚焦"视觉强提示

  目前仅用 .current class 的背景色区分（docCommand.vue:2172），在 PDA
  小屏幕上不够明显。建议加一个左侧色条：

  .param-item.current {
    border-left: 6rpx solid var(--color-primary); // 用主题色
    background: rgba(22,119,255,0.05);
  }

  11. 指令按钮列表可改为可搜索的下拉

  当前指令选择有两套 UI：底部大按钮列表（未选时）+ 顶部
  SearchSelector（选择后）。用户可能面对 20+
  个指令时按钮列表体验差。建议统一为：未选时显示搜索下拉选择器 +
  常用指令快捷按钮（最多 4 个）。
